مستندات جامع پروژه مدیریت تیکت (نسخه طولانی)
1) مقدمه و هدف پروژه

این پروژه با هدف پیاده‌سازی یک سامانه‌ی مدیریت تیکت (Ticket Management System) با استفاده از Python و فریم‌ورک قدرتمند Django طراحی و توسعه یافته است. هدف اصلی، ایجاد یک پلتفرم کارآمد برای کاربران جهت ثبت، مشاهده، پیگیری و مدیریت درخواست‌های خود (تیکت‌ها) است. این سیستم به‌گونه‌ای طراحی شده که نقش‌های کاربری مختلف (کاربر عادی، staff، superuser) را در نظر بگیرد و دسترسی‌ها را بر اساس آن‌ها مدیریت کند.
2) تکنولوژی‌های کلیدی

    زبان برنامه‌نویسی: Python (نسخه‌ی سازگار با Django)
    فریم‌ورک: Django (برای مدیریت وب، ORM، مسیریابی و…)
    قالب‌بندی صفحات: Django Templates (برای ایجاد صفحات HTML پویا)
    زبان قالب‌ها: HTML، CSS ( برای ظاهر صفحه)
    مدیریت پایگاه داده: Django ORM (که به طور پیش‌فرض با SQLite یا پایگاه‌های داده دیگر کار می‌کند)
    امنیت: استفاده از مکانیزم‌های داخلی Django مانند login_required و مدیریت session.
    مسیریابی: Django URL routing برای هدایت درخواست‌های HTTP به ویوهای مناسب.

3) معرفی ساختار فایل‌ها و کامپوننت‌ها
3.1) فایل views.py (منطق اصلی برنامه)

این فایل هسته‌ی منطقی برنامه را در خود جای داده و وظیفه‌ی پردازش درخواست‌های کاربر و ارسال پاسخ را بر عهده دارد.

    ticket_list(request):
    عملکرد: نمایش لیستی از تیکت‌ها.
    محدودیت دسترسی: کاربران عادی فقط تیکت‌های خودشان را می‌بینند، در حالی که کاربران با نقش staff یا superuser به همه‌ی تیکت‌ها دسترسی دارند.
    قالب: از tickets/ticket_list.html برای نمایش لیست استفاده می‌کند.
    نام URL: ticket_list (در صورت عدم استفاده از namespace).
    ticket_detail(request, ticket_id):
    عملکرد: نمایش جزئیات کامل یک تیکت مشخص.
    محدودیت دسترسی: بررسی می‌کند که آیا کاربر مجاز به دیدن جزئیات این تیکت هست یا خیر.
    ورودی: پارامتر ticket_id (یا id بسته به تعریف URL).
    قالب: از tickets/ticket_detail.html استفاده می‌کند.
    نام URL: ticket_detail.
    create_ticket(request):
    عملکرد: مدیریت فرآیند ایجاد تیکت جدید.
    روش‌ها: از متدهای GET (برای نمایش فرم خالی) و POST (برای دریافت و ذخیره‌ی داده‌ها) پشتیبانی می‌کند.
    فرم: از TicketForm برای اعتبارسنجی داده‌ها استفاده می‌کند.
    قالب: از tickets/create_ticket.html برای نمایش فرم استفاده می‌کند.
    پس از ثبت: کاربر به صفحه لیست تیکت‌ها هدایت می‌شود.
    نام URL: ticket_create.
    update_ticket(request, ticket_id):
    عملکرد: مدیریت فرآیند ویرایش یک تیکت موجود.
    ورودی: پارامتر ticket_id (یا id).
    روش‌ها: GET (برای نمایش فرم با اطلاعات فعلی) و POST (برای ذخیره‌ی تغییرات).
    فرم: از TicketForm استفاده می‌کند.
    قالب: از tickets/update_ticket.html استفاده می‌کند.
    پس از ویرایش: کاربر به صفحه جزئیات تیکت هدایت می‌شود.
    نام URL: ticket_update.

3.2) فایل ticket_list.html (قالب نمایش لیست)

این فایل مسئول نمایش بصری لیست تیکت‌ها است و از قوانین ارث‌بری قالب‌های Django بهره می‌برد.

    ارث‌بری: از base.html ارث‌بری می‌کند ({% extends "base.html" %}).
    بارگذاری تگ‌ها: برای استفاده از قابلیت‌های بین‌المللی‌سازی (i18n) از {% load i18n %} استفاده می‌کند.
    تعریف بلوک‌ها: معمولاً بلوک‌هایی مانند title و content را برای قرار دادن محتوای پویا پر می‌کند.
    نمایش دکمه ایجاد تیکت: این دکمه فقط برای کاربران staff یا superuser نمایش داده می‌شود (با استفاده از {% if request.user.is_staff or request.user.is_superuser %}).
    جدول تیکت‌ها: در صورت وجود تیکت، اطلاعات اصلی در قالب یک جدول HTML نمایش داده می‌شوند:
    Subject
    Status
    Priority
    Assigned To
    Created At
    لینک‌های عملیاتی: برای هر تیکت، لینک‌هایی برای مشاهده جزئیات ({% url 'tickets:ticket_detail' ticket.id %}) و ویرایش ({% url 'tickets:ticket_update' ticket.id %}) تعبیه شده است. نکته مهم: استفاده از namespace tickets: ضروری است.
    پیام عدم وجود تیکت: در صورتی که لیستی از تیکت‌ها وجود نداشته باشد، پیامی مانند No tickets found. نمایش داده می‌شود.

3.3) فایل base.html (قالب پایه)

این فایل به عنوان قالب اصلی یا والد برای سایر قالب‌ها عمل می‌کند و ساختار کلی صفحات (مانند سربرگ، پاورقی، لینک‌های CSS و JS) را تعریف می‌کند. سایر قالب‌ها با استفاده از {% extends "base.html" %} به آن متصل می‌شوند.
3.4) فایل eror.txt و New Text Document.txt

این فایل‌ها حاوی خطاهای گزارش‌شده توسط کاربر هستند. تحلیل این خطاها (مانند RuntimeError: Conflicting 'ticket' models... یا Template error: Reverse for 'ticket_create' not found...) به درک مشکلات و رفع آن‌ها کمک شایانی کرده است.
3.5) فایل image.png

یک فایل تصویری که احتمالاً بخشی از رابط کاربری یا مستندات بصری پروژه است.
3.6) فایل views.py (نسخه دوم)

وجود دو فایل views.py در لیست فایل‌ها ممکن است نشان‌دهنده مراحل مختلف توسعه یا خطای در سازماندهی فایل‌ها باشد. نسخه دوم احتمالاً نسخه‌ی اصلاح‌شده یا نهایی‌تر است که در آن برخی خطاها (مانند تعریف تکراری مدل‌ها یا فرم‌ها) برطرف شده‌اند.
3.7) manage.py

برای راه انداختن سرور مجازی روی کامپیوترتون شما باید این دستور هارو بزنید ابتیدا ماشین مجازی (venv) رو با این دستور فعال کنید :
venv\Scripts\activate
بعدش هم با این دستور سرور رو ران کنید و لذت ببرید : 
python manage.py runserver
اگر به مشکلی خوردید حتما هم این کارو تو cmd.exe بکنید


4) قوانین طلایی کدنویسی و استفاده از Django
4.1) اصل جداسازی دغدغه‌ها (Separation of Concerns)

    مدل‌ها فقط در models.py: تعریف کلاس‌های مدل (برای ساختار داده‌ها) باید منحصراً در فایل models.py صورت گیرد. تکرار آن‌ها در views.py یا سایر فایل‌ها باعث خطا (مانند Conflicting 'ticket' models) می‌شود.
    ویوها در views.py: منطق پردازش درخواست‌ها و پاسخ‌دهی باید در فایل views.py پیاده‌سازی شود.
    فرم‌ها در forms.py: اعتبارسنجی ورودی‌های کاربر باید از طریق کلاس‌های فرم که در forms.py تعریف می‌شوند، انجام شود.

4.2) مدیریت URLها و نام‌گذاری (URL Naming and Namespacing)

    نام‌گذاری دقیق: هر مسیر URL باید یک نام منحصربه‌فرد داشته باشد تا بتوان از آن در قالب‌ها یا ویوهای دیگر ارجاع داد (مثلاً name='ticket_list').
    استفاده از Namespace: در پروژه‌های بزرگتر، برای جلوگیری از تداخل نام URLها بین اپلیکیشن‌های مختلف، از namespace استفاده می‌شود. اگر در urls.py اپلیکیشن، app_name = 'tickets' تعریف شده باشد، برای ارجاع به URLها باید از فرمت {% url 'tickets:url_name' %} استفاده کرد (مثلاً {% url 'tickets:ticket_create' %}). عدم رعایت این مورد منجر به خطای NoReverseMatch می‌شود.
    پارامترهای URL: نام پارامترهایی که در مسیر URL تعریف می‌شوند (مانند <int:id>) باید دقیقاً با نام پارامترهای تابع ویو مطابقت داشته باشند (مثلاً def ticket_detail(request, id):).

4.3) امنیت و احراز هویت

    @login_required: برای محافظت از ویوهایی که نیاز به ورود کاربر دارند، باید از این دکوراتور استفاده کرد. این امر اطمینان می‌دهد که فقط کاربران وارد شده به این صفحات دسترسی دارند.
    کنترل دسترسی بر اساس نقش: در ویوهایی مانند ticket_list، دسترسی به داده‌ها بر اساس نقش کاربر (staff/superuser) فیلتر می‌شود تا امنیت و محرمانگی اطلاعات حفظ گردد.

4.4) سازگاری اجزا

    تطابق قالب با مدل/فرم: نام فیلدهایی که در قالب HTML نمایش داده می‌شوند (مانند subject, status, priority) باید با نام فیلدهای تعریف شده در مدل (models.py) و فرم (forms.py) مطابقت داشته باشند. عدم تطابق می‌تواند منجر به خطاهای نمایش یا عملکرد نادرست شود.

5) خلاصه و نتیجه‌گیری

این پروژه با رعایت اصول مهندسی نرم‌افزار و استفاده بهینه از قابلیت‌های Django، یک سامانه‌ی مدیریت تیکت کارآمد را پیاده‌سازی کرده است. ساختار منظم فایل‌ها، مدیریت صحیح URLها، و رعایت قوانین کدنویسی، این پروژه را برای توسعه، نگهداری و افزودن امکانات جدید آماده ساخته است.

ساخت یوزر جدید در Django با CMD

برای ساخت یوزر جدید در پروژه Django، دو روش رایج وجود دارد:
روش اول: ساخت ادمین

اگر بخواهی یک کاربر مدیر بسازی، این دستور را در CMD یا ترمینال اجرا کن:

                                                                    bash
python manage.py createsuperuser

بعد از اجرای دستور:

    نام کاربری را وارد کن
    ایمیل را وارد کن
    رمز عبور را وارد کن
    دوباره رمز عبور را تأیید کن

این کاربر می‌تواند وارد پنل مدیریت Django شود.
روش دوم: ساخت کاربر معمولی

اگر بخواهی یک کاربر عادی بسازی، ابتدا وارد shell جنگو شو:

                                                                    bash
python manage.py shell

سپس این کدها را وارد کن:

                                                                    python
from django.contrib.auth.models import User

user = User.objects.create_user(
    username='ali',
    password='12345678',
    email='ali@example.com'
)
user.save()

توضیح خط به خط

    from django.contrib.auth.models import Userمدل User را از جنگو وارد می‌کند.
    create_user(...)یک کاربر جدید می‌سازد.
    user.save()اطلاعات کاربر را در دیتابیس ذخیره می‌کند
--اگر مشکل داشتی ابتدا با این روش مشکل خود رو حل کن به cmd.exe برو بعدش اینو بزن تا کتابخونه مجازی (venv) فعالشه 
venv\Scripts\activate
