وثائق المشروع الشاملة: نظام إدارة التذاكر (النسخة الموسعة)
1) مقدمة وهدف المشروع

تم تصميم هذا المشروع لتنفيذ نظام إدارة التذاكر باستخدام Python وإطار العمل القوي Django. الهدف الأساسي هو إنشاء منصة فعالة للمستخدمين لتسجيل طلباتهم (التذاكر)، وعرضها، وتتبعها، وإدارتها. تم تصميم النظام مع مراعاة أدوار المستخدمين المختلفة (مستخدم عادي، موظف، مشرف) وإدارة الوصول وفقًا لذلك.
2) التقنيات الرئيسية

    لغة البرمجة: Python (إصدار متوافق مع Django)
    إطار العمل: Django (لإدارة الويب، ORM، التوجيه، إلخ)
    محرك القوالب: Django Templates (لإنشاء صفحات HTML ديناميكية)
    لغة القوالب: HTML، CSS (يُحتمل استخدامها لتصميم الصفحة)
    إدارة قواعد البيانات: Django ORM (يعمل افتراضيًا مع SQLite أو قواعد بيانات أخرى)
    الأمان: استخدام آليات Django المدمجة مثل login_required وإدارة الجلسات.
    التوجيه: توجيه URL الخاص بـ Django لتوجيه طلبات HTTP إلى العروض (views) المناسبة.

3) نظرة عامة على هيكل الملفات والمكونات
3.1) views.py (منطق التطبيق الأساسي)

يحتوي هذا الملف على منطق التطبيق الأساسي المسؤول عن معالجة طلبات المستخدم وإرسال الاستجابات.

    ticket_list(request):
    الوظيفة: عرض قائمة بالتذاكر.
    التحكم في الوصول: يرى المستخدمون العاديون تذاكرهم الخاصة فقط، بينما يتمتع الموظفون أو المشرفون بالوصول إلى جميع التذاكر.
    القالب: يستخدم tickets/ticket_list.html لعرض القائمة.
    اسم URL: ticket_list (إذا لم يتم استخدام مساحة أسماء).
    ticket_detail(request, ticket_id):
    الوظيفة: عرض التفاصيل الكاملة لتذكرة معينة.
    التحكم في الوصول: يتحقق مما إذا كان المستخدم مصرحًا له بعرض تفاصيل التذكرة.
    الإدخال: المعامل ticket_id (أو id، اعتمادًا على تكوين URL).
    القالب: يستخدم tickets/ticket_detail.html.
    اسم URL: ticket_detail.
    create_ticket(request):
    الوظيفة: إدارة عملية إنشاء تذكرة جديدة.
    الطرق: يدعم GET (لعرض نموذج فارغ) و POST (لتلقي البيانات وحفظها).
    النموذج: يستخدم TicketForm للتحقق من صحة البيانات.
    القالب: يستخدم tickets/create_ticket.html لعرض النموذج.
    بعد الإرسال: يتم إعادة توجيه المستخدم إلى صفحة قائمة التذاكر.
    اسم URL: ticket_create.
    update_ticket(request, ticket_id):
    الوظيفة: إدارة عملية تحرير تذكرة موجودة.
    الإدخال: المعامل ticket_id (أو id).
    الطرق: GET (لعرض النموذج مع البيانات الحالية) و POST (لحفظ التغييرات).
    النموذج: يستخدم TicketForm.
    القالب: يستخدم tickets/update_ticket.html.
    بعد التحرير: يتم إعادة توجيه المستخدم إلى صفحة تفاصيل التذكرة.
    اسم URL: ticket_update.

3.2) ticket_list.html (قالب عرض القائمة)

هذا الملف مسؤول عن التمثيل المرئي لقائمة التذاكر ويستفيد من وراثة قوالب Django.

    الوراثة: يرث من base.html ({% extends "base.html" %}).
    تحميل الوسوم: يقوم بتحميل {% load i18n %} لميزات التدويل (Internationalization).
    تعريف الكتل: عادةً ما يملأ كتل مثل title و content لمحتوى الصفحة الديناميكي.
    عرض زر “إنشاء تذكرة”: يكون هذا الزر مرئيًا فقط لحسابات الموظفين أو المشرفين المصادق عليهم (باستخدام {% if request.user.is_staff or request.user.is_superuser %}).
    جدول التذاكر: إذا كانت هناك تذاكر موجودة، يتم عرض معلوماتها الرئيسية في جدول HTML:
    Subject (الموضوع)
    Status (الحالة)
    Priority (الأولوية)
    Assigned To (المعين له)
    Created At (تاريخ الإنشاء)
    روابط الإجراءات: لكل تذكرة، يتم توفير روابط لعرض التفاصيل ({% url 'tickets:ticket_detail' ticket.id %}) والتحرير ({% url 'tickets:ticket_update' ticket.id %}). ملاحظة هامة: استخدام مساحة الاسم tickets: إلزامي.
    رسالة “لا توجد تذاكر”: في حالة عدم توفر تذاكر، يتم عرض رسالة مثل No tickets found. (لم يتم العثور على تذاكر).

3.3) base.html (القالب الأساسي)

يعمل هذا الملف كقالب رئيسي أو أب للقوالب الأخرى، ويحدد الهيكل العام للصفحات (مثل الرأس، والتذييل، وروابط CSS/JS). تتصل القوالب الأخرى به باستخدام {% extends "base.html" %}.
3.4) eror.txt و New Text Document.txt

تحتوي هذه الملفات على سجلات الأخطاء التي أبلغ عنها المستخدم. كان لتحليل هذه الأخطاء (مثل RuntimeError: Conflicting 'ticket' models... أو Template error: Reverse for 'ticket_create' not found...) دورًا حاسمًا في فهم المشكلات وحلها.
3.5) image.png

ملف صورة، من المحتمل أن يكون جزءًا من واجهة المستخدم أو التوثيق المرئي للمشروع.
3.6) views.py (الإصدار الثاني)

قد يشير وجود ملفي views.py في القائمة إلى مراحل تطوير مختلفة أو مشكلة في تنظيم الملفات. من المحتمل أن يكون الإصدار الثاني نسخة مصححة أو نهائية تم فيها حل بعض الأخطاء (مثل تعريفات النماذج أو النماذج المكررة).

3.7) manage.py 
برای راه انداختن سرور مجازی روی کامپیوترتون شما باید این دستور هارو بزنید ابتیدا ماشین مجازی (venv) رو با این دستور فعال کنید :
venv\Scripts\activate
بعدش هم با این دستور سرور رو ران کنید و لذت ببرید : 
python manage.py runserver
اگر به مشکلی خوردید حتما هم این کارو تو cmd.exe بکنید

4) القواعد الذهبية للترميز واستخدام Django
4.1) مبدأ فصل الاهتمامات (Separation of Concerns)

    النماذج فقط في models.py: يجب أن يتم تعريف فئات النماذج (لبنية البيانات) حصريًا في models.py. إعادة تعريفها في views.py أو ملفات أخرى سيؤدي إلى أخطاء (مثل Conflicting 'ticket' models).
    العروض في views.py: يجب تنفيذ منطق معالجة الطلبات والاستجابة في views.py.
    النماذج في forms.py: يجب التعامل مع التحقق من صحة إدخالات المستخدم عبر فئات النماذج المعرفة في forms.py.

4.2) إدارة أسماء وعناوين URL (URL Naming and Namespacing)

    التسمية الدقيقة: يجب أن يكون لكل مسار URL اسم فريد للإشارة إليه في القوالب أو العروض الأخرى (مثل name='ticket_list').
    استخدام مساحات الأسماء: في المشاريع الأكبر، تُستخدم مساحات الأسماء لتجنب تضارب أسماء URL بين التطبيقات المختلفة. إذا تم تعريف app_name = 'tickets' في urls.py الخاص بالتطبيق، فيجب الإشارة إلى عناوين URL باستخدام الصيغة {% url 'tickets:url_name' %} (مثل {% url 'tickets:ticket_create' %}). عدم القيام بذلك يؤدي إلى خطأ NoReverseMatch.
    معاملات URL: يجب أن تتطابق أسماء المعاملات المعرفة في مسار URL (مثل <int:id>) تمامًا مع أسماء المعاملات في دالة العرض المقابلة (مثل def ticket_detail(request, id):).

4.3) الأمان والمصادقة

    @login_required: يجب استخدام هذا المُزخرف (decorator) لحماية العروض التي تتطلب تسجيل دخول المستخدم، مما يضمن وصول المستخدمين المصادق عليهم فقط إليها.
    التحكم في الوصول المستند إلى الدور: في العروض مثل ticket_list، يتم تصفية الوصول إلى البيانات بناءً على دور المستخدم (موظف/مشرف) للحفاظ على الأمان وسرية المعلومات.

4.4) توافق المكونات

    اتساق القالب مع النموذج/النموذج: يجب أن تتماشى أسماء الحقول المعروضة في قوالب HTML (مثل subject, status, priority) مع أسماء الحقول المعرفة في النماذج (models.py) والنماذج (forms.py). قد يؤدي عدم الاتساق إلى أخطاء في العرض أو خلل في الأداء.

5) الملخص والخاتمة

ينفذ هذا المشروع بنجاح نظام إدارة تذاكر فعال من خلال الالتزام بمبادئ هندسة البرمجيات والاستفادة من قدرات Django. تجعل بنية الملفات المنظمة، والإدارة الصحيحة لعناوين URL، والالتزام بمعايير الترميز هذا المشروع جاهزًا لمزيد من التطوير وإضافة الميزات.

العربية
إنشاء مستخدم جديد في Django باستخدام CMD

لإنشاء مستخدم جديد في مشروع Django، توجد طريقتان شائعتان:
الطريقة الأولى: إنشاء مستخدم مدير

إذا كنت تريد إنشاء مستخدم إداري، شغّل هذا الأمر في CMD أو الطرفية:

                                                                    bash
python manage.py createsuperuser

بعد تنفيذ الأمر:

    أدخل اسم المستخدم
    أدخل البريد الإلكتروني
    أدخل كلمة المرور
    أكد كلمة المرور مرة أخرى

هذا المستخدم يمكنه الدخول إلى لوحة تحكم Django.
الطريقة الثانية: إنشاء مستخدم عادي

إذا كنت تريد إنشاء مستخدم عادي، افتح أولاً shell الخاص بـ Django:

                                                                    bash
python manage.py shell

ثم اكتب الكود التالي:

                                                                    python
from django.contrib.auth.models import User

user = User.objects.create_user(
    username='ali',
    password='12345678',
    email='ali@example.com'
)
user.save()

شرح السطور

    from django.contrib.auth.models import Userيستورد نموذج المستخدم المدمج في Django.
    create_user(...)ينشئ حساب مستخدم جديد.
    user.save()يحفظ المستخدم في قاعدة البيانات.

ملاحظات مهمة

    من الأفضل استخدام كلمة مرور قوية.
    إذا كان اسم المستخدم موجودًا مسبقًا، سيظهر خطأ.
    تأكد أنك داخل مجلد المشروع قبل تنفيذ الأوامر.
    إذا كان المشروع يستخدم قاعدة بيانات، فيجب أن تكون عمليات الترحيل قد نُفذت مسبقًا.
